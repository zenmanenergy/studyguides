AP Computer Science Principles -- Unit 4: Algorithms

What Is an Algorithm?

An algorithm is a precise, finite set of instructions for solving a problem or accomplishing a task. Algorithms must eventually terminate and produce a result.

Key properties of a correct algorithm: (1) It must be finite -- it terminates after a defined number of steps. (2) It must be unambiguous -- each step has exactly one interpretation. (3) It must be correct -- it produces the right output for all valid inputs.

Algorithms can be expressed as:

  • Natural language (English steps)
  • Pseudocode
  • Flowcharts
  • Actual programming language code

Sequential, Selection, and Iteration

All algorithms are built from three fundamental structures:

StructureDescriptionExample
SequenceSteps executed one after another in orderStep 1, Step 2, Step 3...
SelectionA decision point: execute different code depending on a conditionIF/ELSE statements
IterationRepeating a block of stepsREPEAT, FOR EACH loops

Linear Search

Linear search checks each element of a list one at a time until the target is found (or the list ends).

PROCEDURE linearSearch(list, target) { FOR EACH item IN list { IF (item = target) { RETURN true } } RETURN false }
  • Best case: target is the first element -- 1 comparison
  • Worst case: target is not in the list -- N comparisons (N = list length)
  • Works on any list, sorted or unsorted

Binary Search

Binary search works only on sorted lists. It repeatedly halves the search space by comparing the target to the middle element.

// Assumes list is sorted in ascending order PROCEDURE binarySearch(list, target) { low <-- 1 high <-- LENGTH(list) REPEAT UNTIL (low > high) { mid <-- (low + high) / 2 IF (list[mid] = target) { RETURN mid } ELSE IF (list[mid] < target) { low <-- mid + 1 } ELSE { high <-- mid - 1 } } RETURN -1 }
Binary search is MUCH faster than linear search for large lists. For a list of 1,000,000 items: linear search may check all 1,000,000; binary search checks at most 20 (log2 of 1,000,000 is about 20).

Sorting Algorithms

Sorting organizes a list into a defined order. AP CSP expects conceptual understanding of common sorting approaches.

AlgorithmHow It WorksEfficiency (rough)
Selection sortFind the minimum, swap it to the front; repeat for the restSlow for large lists (N^2)
Insertion sortTake each element and insert it into its correct position in the sorted portionSlow for large lists; fast for nearly-sorted data
Merge sortDivide list in half repeatedly, sort each half, merge back togetherMuch faster (N log N); used in practice

Algorithmic Efficiency

Efficiency describes how an algorithm's resource usage (time, memory) grows as the input size grows. We care most about behavior on large inputs.

Growth TypeDescriptionExample
Constant O(1)Same time regardless of input sizeAccess list element by index
Logarithmic O(log n)Doubles input = one more stepBinary search
Linear O(n)Steps grow proportionally to inputLinear search, traverse a list
Quadratic O(n^2)Steps grow with square of inputNested loops; selection sort
Exponential O(2^n)Doubles with each additional input elementSome brute-force problems; quickly impractical
An algorithm that is "reasonable" runs in polynomial time (O(n^2) or better). An "unreasonable" algorithm runs in exponential time -- for large inputs, these are effectively unsolvable, even with the fastest computers.

Undecidable and Unsolvable Problems

Not every problem can be solved by an algorithm. Some problems are undecidable -- there is no algorithm that will always produce a correct yes/no answer.

  • The Halting Problem: Given a program and its input, will the program eventually halt (stop running)? There is no algorithm that can determine this for all programs. Proven undecidable by Alan Turing.
  • Undecidable does not mean we can never get an answer for specific cases -- it means no single algorithm can solve all cases
  • Some problems are decidable but unreasonably slow -- a correct algorithm exists, but it would take longer than the age of the universe to run on large inputs
